## Overview

The `underpost push` command pushes committed changes from a local repository to a remote GitHub repository. It integrates with Underpost's authentication system and supports force pushing and custom extensions.

## Usage

```bash
underpost push <path> <uri> [options]
```

## Arguments

<ParamField path="path" type="string" required>
  The absolute or relative directory path of the repository.
  
  **Example:** `./` or `../my-project`
</ParamField>

<ParamField path="uri" type="string" required>
  The URI of the GitHub repository in the format `username/repository`.
  
  **Example:** `underpostnet/pwa-microservices-template`
</ParamField>

## Options

<ParamField path="-f" type="boolean" default="false">
  Forces the push, overwriting the remote repository history.
  
  **Warning:** Use with caution as this can overwrite remote changes and cause issues for other collaborators.
</ParamField>

<ParamField path="-g8" type="boolean" default="false">
  Uses the `.g8` repository extension instead of `.git` for pushing.
  
  This is a custom extension used in the Underpost ecosystem.
</ParamField>

## Environment Variables

The push command requires the following environment variables:

- **GITHUB_TOKEN**: Required for authentication. Must have push access to the repository.
- **GITHUB_USERNAME**: Used as the default username if not specified in the URI.

## Examples

### Basic Push

Push changes to the remote repository:

```bash
underpost push ./ underpostnet/engine
```

After pushing, displays the commit URL:
```
commit url http://github.com/underpostnet/engine/commit/abc123...
```

### Push from Specific Directory

Push changes from a specific repository directory:

```bash
underpost push ../my-project myusername/my-project
```

### Force Push

Force push to overwrite remote history:

```bash
underpost push ./ underpostnet/engine -f
```

**Warning:** Only use force push when absolutely necessary, such as:
- After rebasing local commits
- When you need to remove sensitive data
- When working on a personal branch

### Push with G8 Extension

Push to a repository using the `.g8` extension:

```bash
underpost push ./ underpostnet/engine -g8
```

### Push to Private Repository

Ensure your `GITHUB_TOKEN` has appropriate permissions:

```bash
export GITHUB_TOKEN=ghp_your_token_here
underpost push ./ yourusername/private-repo
```

## Complete Workflow

### Standard Development Workflow

```bash
# 1. Clone the repository
underpost clone underpostnet/engine

# 2. Navigate to repository
cd engine

# 3. Make changes to files
vim src/index.js

# 4. Stage changes
git add .

# 5. Commit with conventional format
underpost cmt ./ feat "Add new feature"

# 6. Push to remote
underpost push ./ underpostnet/engine
```

### Feature Branch Workflow

```bash
# Create and checkout feature branch
git checkout -b feature/new-api

# Make changes and commit
vim src/api.js
git add .
underpost cmt ./ feat api "Add new endpoint"

# Push feature branch
git push origin feature/new-api

# Or use underpost push
underpost push ./ underpostnet/engine
```

### Hotfix Workflow

```bash
# Pull latest changes
underpost pull ./ underpostnet/engine

# Fix the bug
vim src/bug.js
git add .
underpost cmt ./ fix "Resolve critical bug"

# Push immediately
underpost push ./ underpostnet/engine
```

## Output

After successfully pushing, the command displays:

1. **Commit URL**: Direct link to view the commit on GitHub
   ```
   commit url http://github.com/username/repo/commit/abc123def456...
   ```

This allows you to quickly navigate to the commit on GitHub to:
- Verify the changes
- Share the commit with team members
- Create a pull request
- Check CI/CD status

## Behavior

- Pushes to the current branch of the remote repository
- Uses HTTPS protocol for pushing
- Requires `GITHUB_TOKEN` for authentication
- Displays the commit URL after successful push
- Logs are disabled by default for cleaner output

## Error Handling

### Common Errors

**Authentication Failed**
```bash
fatal: Authentication failed
```
Solution: Ensure `GITHUB_TOKEN` is set and has push permissions.

**Push Rejected**
```bash
error: failed to push some refs
```
Solution: Pull latest changes first with `underpost pull`, then push again.

**No Commits to Push**
```bash
Everything up-to-date
```
This means your local branch is already synced with the remote.

## Best Practices

### Before Pushing

1. **Review your changes**: Use `underpost cmt --diff` to see what you're committing
2. **Pull first**: Run `underpost pull` to avoid conflicts
3. **Test locally**: Ensure your changes work before pushing
4. **Write clear commit messages**: Use conventional commit format

### When to Force Push

✅ **Safe scenarios:**
- Personal feature branches
- After interactive rebase on your branch
- Fixing your own mistakes before others pull

❌ **Avoid force pushing:**
- Main/master branches
- Shared feature branches
- After others have pulled your changes
- Production branches

### Security

- Never commit `GITHUB_TOKEN` to the repository
- Store tokens in environment variables or secure vaults
- Use tokens with minimal required permissions
- Rotate tokens regularly

## Integration with CI/CD

After pushing, automated workflows may trigger:

```bash
# Push triggers CI/CD pipeline
underpost push ./ underpostnet/engine

# Monitor the commit URL for:
# - Build status
# - Test results  
# - Deployment status
```

## Notes

- The command assumes you have already committed your changes locally
- Make sure you're on the correct branch before pushing
- For collaborative projects, consider using pull requests instead of pushing directly to main branches
- The commit URL format works for both standard `.git` and custom `.g8` repositories
